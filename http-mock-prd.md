# HTTP Mock 服务 - 产品需求文档（PRD）

## 1. 项目信息

| 项目 | 内容 |
|------|------|
| **项目名称** | `http_mock_server` |
| **文档版本** | v1.0 |
| **撰写人** | 产品经理 许清楚（Xu） |
| **编程语言/框架** | Python 3.6.8 + Flask（兼容 CentOS 7.9） |
| **目标用户** | 测试工程师（使用 MeterSphere 自动化测试平台） |

### 原始需求复述

测试工程师需要一个功能完善的 HTTP 挡板（Mock Server），用于接口自动化测试场景。核心能力包括：通过改域名接入（不改业务代码）拦截请求、请求数据入库存储并支持按请求标识反查、同接口按入参多规则差异化返回并支持场景切换、自定义响应/延迟/超时/随机异常、以及无匹配规则时转发真实接口的代理模式，同时提供 Web 配置后台管理规则与请求记录。

## 2. 产品目标

| # | 产品目标 | 衡量标准 |
|---|---------|---------|
| G1 | **零侵入接入**：被测系统仅需修改域名指向 Mock 服务，无需改动任何业务代码，即可完成请求拦截与 Mock | 接入步骤 ≤ 2 步（配置域名 + 启动服务）；支持所有标准 HTTP 方法 |
| G2 | **可断言的请求存证**：每次请求自动入库，生成唯一请求标识，测试脚本可凭标识反查完整请求数据用于断言 | 请求记录 100% 落库；查询 API 响应时间 < 200ms；数据包含 Header/Query/Body 全量字段 |
| G3 | **灵活的差异化 Mock**：同一接口可根据入参匹配多规则返回不同响应，支持场景切换、延迟/超时/异常模拟及无匹配转发 | 单接口支持 ≥ 10 条规则；场景切换 < 1s 生效；转发成功率 ≥ 99% |

## 3. 用户故事

| # | 角色 | 需求 | 价值 |
|---|------|------|------|
| US1 | 测试工程师 | 我希望只需将被测系统的接口域名指向 Mock 服务地址，无需修改业务代码就能拦截请求 | 降低接入成本，避免代码侵入，快速搭建测试环境 |
| US2 | 测试工程师 | 我希望每次请求都被自动记录（Header、Query、Body、唯一请求标识），并能通过 API 凭标识反查完整请求数据 | 在 MeterSphere 自动化脚本中拿到请求标识后，可对实际发送的请求数据做精确断言 |
| US3 | 测试工程师 | 我希望同一接口能根据不同的入参（Header/Query/Body 字段）匹配不同规则，返回不同的 Mock 响应 | 用一个 Mock 服务覆盖同一接口的多种测试场景（如正常/异常/边界值） |
| US4 | 测试工程师 | 我希望能预设多个场景并快速切换，切换后命中不同的规则集 | 在测试流程中按需切换场景（如"正常流程"→"异常流程"），无需逐条修改规则 |
| US5 | 测试工程师 | 我希望能配置自定义响应体、模拟延迟、模拟超时、随机返回异常状态码 | 测试被测系统在网络异常、慢响应、服务端错误等极端场景下的容错能力 |
| US6 | 测试工程师 | 我希望当请求未命中任何 Mock 规则时，能自动转发到真实后端接口 | 部分接口走 Mock、部分接口走真实服务，混合测试更灵活 |

## 4. 需求池（Requirements Pool）

### P0 — MVP 必须（Must Have）

| ID | 需求名称 | 描述 | 验收标准 |
|----|---------|------|---------|
| P0-1 | **多方法请求拦截** | 支持 GET/POST/PUT/DELETE/PATCH 等标准 HTTP 方法的请求拦截与响应 | 覆盖 ≥ 6 种 HTTP 方法；任意方法请求均能返回 Mock 响应 |
| P0-2 | **域名接入** | Mock 服务以独立 HTTP 服务运行，被测系统修改域名/Host 即可接入，不改业务代码 | 服务启动后监听指定端口；支持任意 Host 头请求 |
| P0-3 | **请求数据入库** | 每次请求自动将 Header、Query、Body、HTTP 方法、URL 路径、时间戳存入数据库，并生成唯一请求标识（Request ID） | 请求到达后 100% 落库；Request ID 全局唯一；数据完整无丢失 |
| P0-4 | **请求查询 API** | 开放 HTTP API，凭 Request ID 查询本次请求的完整数据（Header/Query/Body 等），供 MeterSphere 自动化脚本调用做断言 | `GET /api/requests/{request_id}` 返回完整请求数据；响应 JSON 格式；响应时间 < 200ms |
| P0-5 | **Mock 规则匹配** | 支持按 URL + HTTP 方法 + 请求参数（Header/Query/Body 字段值）匹配规则，返回预设响应 | 支持精确匹配与字段值匹配；命中规则后返回对应响应体、状态码、Header |
| P0-6 | **自定义响应配置** | 可为每条规则配置自定义响应体（JSON/文本）、HTTP 状态码、响应 Header | 支持配置任意状态码；响应体支持 JSON 与纯文本；Header 可自定义 |
| P0-7 | **Web 配置后台** | 提供 Web 管理界面，支持规则增删改查、请求记录查看 | 可通过浏览器管理规则；可查看请求记录列表与详情 |
| P0-8 | **规则匹配引擎** | 请求到达时按优先级依次匹配规则，命中第一条即返回 | 支持规则优先级排序；未命中时返回默认响应（404 或可配置） |

### P1 — 应该有（Should Have）

| ID | 需求名称 | 描述 | 验收标准 |
|----|---------|------|---------|
| P1-1 | **场景管理** | 支持创建多个场景（如"正常流程""异常流程"），每个场景关联一组规则，可一键切换当前生效场景 | 场景切换后 < 1s 生效；同一时刻仅一个场景生效（或支持多场景并行）；切换不影响其他配置 |
| P1-2 | **延迟模拟** | 可为规则配置响应延迟时间（毫秒），模拟慢响应 | 配置延迟后实际响应延迟误差 < 100ms；支持固定延迟值 |
| P1-3 | **超时模拟** | 可为规则配置超时行为（不返回响应，直到客户端超时） | 配置超时后 Mock 服务不返回响应；被测系统按自身超时配置断开连接 |
| P1-4 | **随机异常** | 可为规则配置按概率随机返回异常状态码（如 50% 返回 500、50% 返回 200） | 支持配置概率百分比；多次请求的异常率符合预期分布 |
| P1-5 | **无匹配转发** | 当请求未命中任何 Mock 规则时，可配置转发到真实后端地址（HTTP 代理模式），将真实响应返回客户端 | 转发保留原始请求方法/Header/Body；支持配置目标地址；转发成功率 ≥ 99% |
| P1-6 | **请求记录筛选** | 请求记录列表支持按 URL、方法、时间范围、场景筛选 | 支持多条件组合筛选；分页查询；响应时间 < 500ms |
| P1-7 | **规则启停** | 支持单条规则的启用/禁用，禁用的规则不参与匹配 | 禁用规则后立即不匹配；启用后立即生效 |

### P2 — 可以有（Nice to Have）

| ID | 需求名称 | 描述 | 验收标准 |
|----|---------|------|---------|
| P2-1 | **规则导入导出** | 支持规则的批量导入导出（JSON 格式），便于环境间迁移 | 导出文件可重新导入恢复完整配置 |
| P2-2 | **请求记录自动清理** | 支持配置请求记录保留天数，超期自动清理 | 可配置保留天数；定时任务自动清理 |
| P2-3 | **正则匹配** | URL 匹配支持正则表达式模式 | 支持正则匹配 URL 路径；性能无明显下降 |
| P2-4 | **请求统计仪表盘** | 配置后台首页展示请求量统计、规则命中分布、场景使用情况 | 可视化图表展示；数据准确实时 |
| P2-5 | **操作日志** | 记录配置后台的操作日志（谁在何时修改了哪条规则） | 操作日志可追溯；支持按操作类型筛选 |
| P2-6 | **多用户与权限** | 支持多用户登录，区分管理员与普通用户权限 | 不同角色看到不同功能范围 |
| P2-7 | **WebSocket Mock** | 支持 WebSocket 协议的 Mock | 可拦截 WS 连接并返回预设消息 |

### 5.2 页面结构

#### 页面一：仪表盘（Dashboard）
- **请求总量统计卡片**：今日请求数、总请求数、Mock 命中数、转发数
- **请求趋势图**：近 7 天请求量折线图
- **规则命中 TOP 10**：命中次数最多的规则排行
- **当前生效场景**：显示当前激活的场景名称

#### 页面二：规则管理（Rules）
- **规则列表表格**：

| 列名 | 说明 |
|------|------|
| 启用 | 开关，控制规则是否参与匹配 |
| 优先级 | 数字，越小越优先匹配 |
| URL | 匹配的 URL 路径 |
| 方法 | HTTP 方法（GET/POST/...） |
| 匹配条件 | 入参匹配规则摘要（如 `body.code == "001"`） |
| 响应摘要 | 状态码 + 响应体前 50 字符 |
| 延迟 | 延迟毫秒数（如有） |
| 所属场景 | 关联的场景名称 |
| 操作 | 编辑 / 删除 / 复制 |

- **新增/编辑规则弹窗**：
  - 基本信息：URL 路径、HTTP 方法、优先级、所属场景
  - 匹配条件：支持添加多条匹配规则（字段来源：Header/Query/Body，字段名，操作符：等于/包含/正则，字段值）
  - 响应配置：状态码、响应 Header（键值对列表）、响应体（JSON 编辑器/文本框）
  - 高级行为：延迟时间（ms）、是否超时不响应、随机异常（概率% + 异常状态码）
- **批量操作**：批量启用/禁用/删除

#### 页面三：场景管理（Scenes）
- **场景列表**：场景名称、关联规则数、状态（生效中/未生效）、创建时间
- **操作**：新建场景、编辑、删除、设为当前生效场景
- **场景切换**：点击"设为当前"一键切换，切换后顶部导航栏显示当前场景名

#### 页面四：请求记录（Requests）
- **筛选栏**：URL（模糊搜索）、HTTP 方法（下拉）、时间范围（日期选择器）、场景（下拉）
- **请求列表表格**：

| 列名 | 说明 |
|------|------|
| 时间 | 请求到达时间 |
| Request ID | 唯一标识（可复制） |
| 方法 | HTTP 方法 |
| URL | 请求路径 |
| 匹配结果 | 命中规则名称 / 转发 / 未匹配 |
| 响应状态码 | 返回的 HTTP 状态码 |
| 耗时 | 处理耗时（ms） |
| 操作 | 查看详情 |

- **请求详情弹窗**：展示完整 Header、Query 参数、Body 内容、匹配的规则信息、返回的响应内容

#### 页面五：设置（Settings）
- **转发配置**：是否启用无匹配转发、默认转发目标地址
- **默认响应**：未命中规则且未启用转发时的默认响应（状态码 + 响应体）
- **数据清理**：请求记录保留天数配置、手动清理按钮
- **服务配置**：监听端口、服务状态（运行中/已停止）

### 5.3 设计风格
- 使用 Material Design 风格，简洁清晰
- 主色调：蓝色系（#1976d2）
- 表格支持分页，默认每页 20 条
- JSON 编辑器带语法高亮
- 关键操作（删除）需二次确认

## 6. 待确认问题（Open Questions）

| # | 问题 | 影响范围 | 建议 |
|---|------|---------|------|
| Q1 | **数据库选型**：使用 SQLite（轻量，无需额外安装）还是 MySQL（适合大量请求记录）？ | 架构设计 | MVP 阶段建议 SQLite，后续可迁移 MySQL |
| Q2 | **场景模式**：同一时刻仅允许一个场景生效，还是支持多场景并行（多场景时规则合并匹配）？ | 场景管理设计 | 建议同一时刻仅一个场景生效，逻辑简单；如需多场景可通过规则启停实现 |
| Q3 | **转发模式细节**：转发时是否需要修改请求 Host 头？是否需要支持 HTTPS 目标？转发请求是否也入库记录？ | 转发功能 | 建议修改 Host 头为目标地址、支持 HTTPS、转发请求同样入库 |
| Q4 | **Request ID 传递方式**：Request ID 是由 Mock 服务自动生成并放入响应 Header 返回给客户端，还是通过其他方式告知测试脚本？ | 查询 API 设计 | 建议自动生成 UUID 放入响应 Header（如 `X-Mock-Request-Id`），测试脚本从响应头提取 |
| Q5 | **请求标识与 MeterSphere 集成**：是否需要 Mock 请求与 MeterSphere 测试用例自动关联，还是仅通过 Request ID 手动查询？ | 自动化集成 | MVP 阶段仅提供查询 API，测试脚本主动查询即可 |
| Q6 | **并发性能要求**：预期同时并发的 Mock 请求量大概是多少？是否需要支持高并发？ | 技术架构 | 需用户确认，影响是否引入异步框架或进程管理 |
| Q7 | **认证需求**：配置后台和查询 API 是否需要鉴权（Token/密码），还是仅在内网使用无需认证？ | 安全设计 | 内网使用建议 MVP 无需鉴权，后续可加 Token |
| Q8 | **现有项目复用**：用户已有的 `/home/jht/http_mock_server` 简单 mock 服务中是否有可复用的代码或配置？ | 开发效率 | 需查看现有代码后再定 |

## 7. 技术约束备忘

| 约束 | 说明 |
|------|------|
| 操作系统 | Linux CentOS 7.9 |
| Python 版本 | 3.6.8（**不可使用 3.7+ 特性**，如 `f-string =` 调试语法、`dict` 有序保证依赖、`asyncio.run()`、`dataclasses` 等 3.7+ 新增 API） |
| Web 框架 | Flask（用户有经验） |
| 前端 | 建议使用轻量方案（如 Bootstrap + 原生 JS，或 Vue 2 CDN 版本），避免复杂构建工具链 |
| 自动化平台 | MeterSphere（查询 API 需兼容 MeterSphere HTTP 请求步骤调用） |
| 部署 | 需兼容 CentOS 7.9 的包管理器（yum）和系统库版本 |
